# react-qa-service 上线检查清单

本文档用于帮助 `react-qa-service` 在上线前完成最小可用、可观测、可回滚的发布准备。

适用范围：

- FastAPI 后端服务
- React + Vite 前端
- Redis 会话与向量存储
- 文档上传、索引、检索、总结、反馈采集链路

---

## 1. 上线目标

建议先明确本次上线属于哪一类：

- 内部试运行：面向小范围真实用户，允许保留少量已知问题，但必须可回滚、可监控。
- 灰度上线：面向部分正式用户，要求核心链路稳定，错误可控。
- 正式上线：面向全部目标用户，要求功能、监控、告警、运维流程完整。

如果当前是第一次对真实用户开放，建议按“内部试运行 -> 灰度上线 -> 正式上线”分阶段推进。

---

## 2. 必须完成

以下项目建议视为上线阻塞项，没有完成前不建议对真实用户开放。

### 2.1 环境与配置

- 确认生产环境 `.env` 或等效配置已准备完成。
- 确认 `openai_api_key`、`openai_base_url`、`llm_model`、`embedding_model` 已配置且可用。
- 确认 `redis_url` 指向正确实例，且容量、持久化、备份策略明确。
- 确认 `jwt_secret` 已替换默认值，不能继续使用弱口令或演示配置。
- 确认 `demo_username` / `demo_password` 不会直接暴露到正式环境，至少应更换默认账号密码。
- 确认前端 `VITE_API_BASE_URL` 指向正确后端地址。

### 2.2 依赖与运行验证

- 在目标环境完整安装 Python 和前端依赖。
- 至少执行一次后端启动验证，确认 `app/main.py` 可以正常启动。
- 至少执行一次前端构建验证，确认页面可正常加载和请求 API。
- 修复当前测试环境依赖缺失问题，确保核心测试可以运行。

建议至少覆盖以下测试或等效验证：

- 认证与登录
- 文档上传与入库
- QA 问答
- Summary 总结
- 反馈提交

### 2.3 核心业务链路验收

必须人工走通以下流程：

1. 登录系统。
2. 上传一个真实文档。
3. 确认文档成功切块并建立索引。
4. 发起一个 QA 问题，确认返回引用片段。
5. 发起一个 Summary 问题，确认进入 summary 链路。
6. 提交一次赞或踩反馈，确认后端成功存储。

验收时至少关注：

- 上传失败时是否有明确报错
- 检索为空时是否能优雅降级
- 总结输出是否具备固定结构
- 前端 inspector 是否正确展示 `task_type`、检索摘要、重排摘要、summary 阶段

### 2.4 安全与访问控制

- 确认认证接口、问答接口、文档接口、反馈接口均受 JWT 保护。
- 确认默认用户、默认密码、测试凭据不会泄露到正式环境。
- 确认跨域配置只允许必要来源。
- 确认限流策略可用，避免接口被高频刷爆。
- 确认上传文件大小限制和文件类型限制符合预期。

### 2.5 监控与日志

- 确认服务启动日志、请求异常日志、外部依赖错误日志可被采集。
- 至少具备以下基础指标：
  - 接口成功率
  - 接口错误率
  - 平均响应时间
  - Redis 连接状态
  - 模型调用失败率
- 确认出现 5xx、Redis 不可用、模型调用失败时能够快速定位。

### 2.6 回滚方案

- 明确本次发布的版本号或 commit SHA。
- 明确失败后的回滚步骤。
- 明确谁负责执行回滚。
- 明确回滚后如何验证服务恢复正常。

如果这四项没有书面化，建议不要直接上线。

---

## 3. 强烈建议完成

这些项目不一定阻塞第一次试运行，但会显著影响上线后的稳定性和排障效率。

### 3.1 真实数据回归

- 准备一批真实或近真实文档样本。
- 覆盖短文档、长文档、Markdown、PDF、DOCX 等常见格式。
- 至少抽样验证以下问题类型：
  - 事实问答
  - 全文总结
  - 风险提炼
  - 连续追问

### 3.2 离线评估执行

仓库中已经有以下脚本：

- `scripts/eval_qa.py`
- `scripts/eval_summary.py`
- `scripts/eval_retrieval.py`

上线前建议至少执行一次，并留存结果，用于和后续版本做对比。

### 3.3 Agent 效果校准

- 检查任务分类误判情况，尤其是 QA 与 Summary 的边界问题。
- 检查 query rewrite 是否引入无关查询。
- 检查 rerank 后是否出现高分但不相关片段。
- 检查 summary 是否因为首轮召回偏差导致漏掉关键章节。

### 3.4 数据与存储治理

- 评估 Redis 中 chunk、session、feedback 的 TTL 和保留策略。
- 明确是否需要清理旧索引与旧会话。
- 明确文档上传后的数据删除策略与责任人。

### 3.5 运维准备

- 确认部署方式固定，例如 Docker Compose、容器平台或现有发布平台。
- 确认重启策略、健康检查、资源限制已配置。
- 确认 Redis、应用服务、前端静态资源各自的运维边界清晰。

---

## 4. 可以延后

这些项目建议纳入后续版本，但不一定阻塞第一阶段试运行。

- 更强的任务分类器，而不是当前偏规则式的 `classify_task`
- 更强的 rerank 模型，而不是当前轻量 fallback 打分
- 更强的 multi-query 改写策略
- `extract` / `compare` 等更多任务链路
- 更完整的线上质量看板和自动告警体系
- 更系统的 benchmark 扩充与自动化评测流水线

---

## 5. 上线前一天检查

建议在上线前一天逐项确认：

- 后端服务可以在目标环境启动
- 前端页面可以正常访问
- 登录功能正常
- 文档上传正常
- QA 正常
- Summary 正常
- 反馈提交正常
- Redis 正常
- 模型调用正常
- 日志可查看
- 回滚方案已确认
- 发布负责人和值班人已确认

---

## 6. 上线当天操作建议

### 发布前

- 冻结主分支高风险变更。
- 记录当前线上版本信息。
- 通知相关人员进入发布窗口。

### 发布中

- 先发布后端，再验证接口。
- 再发布前端，验证页面与接口联调。
- 按照“登录 -> 上传 -> QA -> Summary -> 反馈”的顺序做冒烟测试。

### 发布后 30 分钟内

- 持续观察错误日志和响应时间。
- 重点关注上传、问答、总结、反馈四条主链路。
- 如果错误率明显上升，优先回滚而不是在线修补。

---

## 7. 建议的上线结论模板

可以用下面这个模板做最终判断：

### 可以上线

- 必须完成项全部通过
- 关键链路人工验收通过
- 监控与回滚准备完成
- 已知问题不影响核心功能

### 暂缓上线

- 核心依赖未跑通
- 核心链路存在阻塞性缺陷
- 监控缺失，出问题无法定位
- 回滚方案不明确

---

## 8. 当前项目的建议结论

结合当前代码状态，项目已经接近“可试运行”阶段，但上线前仍建议优先完成以下事项：

1. 补齐并跑通后端依赖，确保核心测试可执行。
2. 用真实文档做一轮完整回归。
3. 检查默认账号密码、JWT 密钥、生产配置是否已替换。
4. 补齐基础监控、日志采集和回滚说明。
5. 以小范围灰度而不是一次性全量开放的方式上线。
